Service Status Dashboard  ·  One real-time view of every service we depend on
 

Hi all,

A quick update on Sentinel, the status-dashboard project. It had been quiet for a couple of months, so I wanted to show where it now stands — the short version is that the build is essentially done.

It pulls the operational health of every external service we depend on into a single, real-time view. Instead of opening a dozen vendor status pages by hand during an incident, the team gets one pane of glass over GitHub, AWS, Cloudflare, Microsoft 365, Salesforce, ICE, Snowflake, Datadog and more — alongside our own custom endpoints.

What’s working today
📊 One pane of glass — 26 status sources (80 monitored endpoints) grouped by area, with a colour-coded banner that tells you at a glance whether anything is degraded.
⚡ Real-time, and kind to rate limits — the server does the polling and pushes changes to every browser over WebSocket. Ten people watching = one call per source, not ten.
🔧 Any source, mostly without code — built-in adapters for Statuspage, AWS, Azure DevOps and RSS/Atom, plus a visual mapping editor (click a node in the JSON to wire it up) for any other API.
🧮 Signal, not noise — composites roll many URLs into one card (e.g. all 49 UK Salesforce instances, the 7 ICE feeds), and noisy feeds like AWS are scoped to the regions and time window we actually care about.
🔐 Secure by design — any per-source API tokens stay on the server (never in the browser), the proxy blocks internal targets (anti-SSRF), and the management pages can be locked behind a password or corporate SSO.
It answers the first question of every incident — “is it them, or is it us?” — in one glance, and it was built entirely in-house.

What’s next

On the build list: proactive alerting — a Teams or email heads-up (via Power Automate) the moment a source crosses a chosen severity, so we hear about an outage before a user does.

The one ask — a hosting home

In all honesty, this is also what has held the project up. Sentinel is a small always-on server — it polls in the background and pushes updates over a live connection — so it can’t simply be “uploaded” somewhere static; it needs somewhere to run. Without a cloud account to host it — ideally a corporate AWS account — it can’t move past a local prototype, which is exactly why it stalled. The footprint is tiny: a single container behind a load balancer, fronted by CloudFront, for roughly $35–80 a month — and it puts the tool under firm ownership, IT security and governance. I’ve written up the full architecture and can share it on request.
